# 开发步骤

## 推进原则

- 一次只完成一个阶段。
- 每个阶段结束后运行可用性检查。
- 每次开发结束更新当天开发日志。
- 阶段外的新想法先写入待办，不直接实现。

## Phase 0：项目标准初始化

目标：

- 创建 `dev-logs/`、`docs/` 和 `AGENTS.md`。
- 写入需求、技术方案、设计规范、开发步骤和数据模型。
- 创建当天开发日志。

验收：

- 所有标准文件存在。
- `AGENTS.md` 正确指向标准文件路径。
- 当天开发日志包含已完成、待办事项、问题/风险、下一步。

## Phase 1：最小桌面应用

目标：

- 初始化 Tauri + React + TypeScript + Vite。
- 实现空白主窗口。
- 添加基础主题变量和基础布局。
- 验证 Windows 下可启动。

验收：

- 应用可以启动。
- 主窗口显示基础界面。
- 无控制台阻断性错误。

## Phase 2：本地数据基础

目标：

- 引入本地 SQLite 或 Tauri 本地数据层。
- 实现记录数据模型。
- 完成新增、读取、编辑、删除能力。

验收：

- 重启后数据仍存在。
- 中文内容保存和读取正常。

## Phase 3：日期归类与普通记录

目标：

- 默认展示今天。
- 支持切换日期。
- 支持新增普通记录并归类到指定日期。

验收：

- 不同日期的数据互不混淆。
- 可补记过去，也可提前记录未来事项。

## Phase 4：to do 事项

目标：

- 支持普通记录和 to do 类型切换。
- to do 右侧添加对勾按钮。
- 完成后保留事项并添加删除线。

验收：

- to do 可在未完成和已完成之间切换。
- 完成事项不被删除。

## Phase 5：逾期滚动

目标：

- 启动应用时检查未完成 to do。
- 过期未完成事项自动移动到今天。
- 滚动后的事项使用红色文字显示。

验收：

- 模拟跨天后，逾期未完成事项出现在今天。
- 已完成事项不滚动。

## Phase 6：搜索与筛选

目标：

- 支持全文搜索。
- 支持全部、普通记录、to do、未完成、已完成筛选。

验收：

- 搜索结果准确。
- 筛选结果符合记录类型和状态。

## Phase 7：Windows 桌面体验

目标：

- 支持系统托盘常驻。
- 支持全局快捷键唤起窗口。
- 支持关闭窗口时最小化到托盘。

验收：

- 托盘菜单可打开主窗口。
- 快捷键可唤起窗口。
- 关闭窗口不会直接退出应用。

## Phase 8：UI 精修与稳定化

目标：

- 按设计规范打磨白色和天蓝色毛玻璃界面。
- 检查中文输入、保存、搜索、重启恢复。
- 补充使用说明和开发日志。

验收：

- 核心流程稳定。
- UI 简洁清爽。
- 文档和日志完整。

## Phase 9：账号与云同步

目标：

- 接入 Supabase Auth，支持邮箱密码登录、注册和退出。
- 保持 SQLite 本地优先，未登录或断网时核心记录能力可用。
- 新增软删除和同步状态字段，支持云端 tombstone。
- 登录后支持启动、登录、保存后的自动同步，以及用户手动同步。
- 同步时只上传本地 `dirty` 记录，避免重复写入全部历史记录。
- 使用 Supabase Postgres + RLS 按用户隔离云端记录。

验收：

- 未配置 Supabase 环境变量时，应用仍可本地使用。
- 未登录时新增、编辑、删除、搜索和重启恢复保持正常。
- 登录后本地现有记录可上传到 Supabase。
- 清空本地数据库或换设备后，登录同一账号可从 Supabase 拉回记录。
- 一端删除记录后，另一端同步后记录不会复活。
- 同一记录多端修改时，`updatedAt` 较新的版本胜出。
- `cargo test`、`npm.cmd test`、`npm.cmd run build` 和 `npm.cmd run tauri -- build` 通过。
